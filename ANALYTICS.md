# Технология аналитики geoanalytics

Документ описывает, **какими методами** проект превращает сырые данные (новости,
котировки, макропоказатели) в аналитику: сводку рынка, отчёт по активу, события,
бэктест и алерты. Это не инструкция по запуску (см. `README.md` / `up.sh`), а описание
самих аналитических подходов и их реализации.

## 0. Принципы

- **Локальный каскад маленьких моделей вместо одной большой.** Железо без дискретной
  GPU и с ограниченной RAM, поэтому тяжёлый универсальный LLM не используется на горячем
  пути. Каждую узкую задачу решает специализированный лёгкий инструмент (NER, сентимент,
  классификатор, эмбеддер), а большой LLM (Qwen2.5-7B через Ollama) подключается только
  для финального связного синтеза и опционален.
- **Чистое ядро отдельно от данных.** Все аналитические алгоритмы — индикаторы,
  корреляции, бэктест, значимость, агрегация свечей — это **чистые функции на списках
  `float`/словарях, без БД и pandas**. Их легко тестировать и они детерминированы.
  DB-раннеры лишь достают данные из Postgres и передают в ядро.
- **Graceful degradation.** Если модель/сеть/LLM недоступны — аналитика не падает, а
  переходит на упрощённый путь (лексикон вместо ruBERT, правила вместо классификатора,
  шаблон вместо LLM, формула вместо обученной значимости).
- **Слои данных.** `raw_documents` (сырые ответы источников) → обработка
  (`processing.py`) → нормализованные таблицы (`articles`, `prices`, `macro_series`,
  `fx_rates`, `events` и т.д.) → аналитические запросы (`query/`).

## 1. Конвейер обогащения новости

Модуль `processing.py`, функция `_process_news`. Для каждой новости последовательно:

1. **Очистка текста** (`nlp/text.py`) — HTML → плоский текст.
2. **Тональность** (`nlp/sentiment.py`) — метка + score ∈ [−1, 1].
3. **Классификация события** (`nlp/classify.py`) — категория из 8 типов.
4. **NER** (`nlp/ner.py`, Natasha) — извлечение ORG/PER/LOC.
5. **Entity-linking** (`nlp/entity_linking.py`) — привязка новости к активам/компаниям.
6. **Значимость** (`nlp/significance.py`) — оценка важности новости ∈ [0, 1].
7. **Фильтр** — незначимый шум не сохраняется.
8. **Эмбеддинг** (`nlp/embeddings.py`) — вектор для семантического поиска (RAG).

### 1.1. Тональность (сентимент)

Score во всех случаях — `P(positive) − P(negative)` ∈ [−1, 1]; метка — argmax из трёх
классов (negative/neutral/positive). Трёхуровневая стратегия (от лучшего к фолбэку):

- **Основной путь (прод)** — **дистиллированная модель**, обученная на «золотой» разметке
  локального LLM-учителя (Qwen2.5): полный fine-tune небольшого энкодера `rubert-tiny2`
  под инвесторскую тональность РФ-финтекста. Подключается через `GEO_SENTIMENT_ADAPTER_PATH`,
  указывающий на каталог полной модели (см. §7). Именно она по умолчанию работает в проде.
- **База** — ruBERT `blanchefort/rubert-base-cased-sentiment` (3 класса): если дистиллят
  не задан/не загрузился.
- **Фолбэк** — лёгкий лексиконный анализатор (словари позитив/негатив), если torch
  недоступен.

`GEO_SENTIMENT_ADAPTER_PATH` может указывать и на LoRA-адаптер, и на полностью дообученную
модель — `nlp/sentiment.py` различает их по содержимому каталога (наличие `adapter_config.json`)
и грузит соответственно, читая порядок меток из `labels.json`. **Важно:** для сентимента
работает именно полный fine-tune, а не LoRA поверх замороженной базы — почему, см. §7.

### 1.2. Классификация события

Категории (`core/types.EventType`): `sanctions, earnings, merger, dividends,
geopolitics, regulation, macro, other`. Два уровня:
- **Основной** — дообученный LoRA-классификатор (`GEO_EVENT_ADAPTER_PATH`).
- **Фолбэк** — упорядоченные регулярные правила по корням слов (от специфичных к общим).

### 1.3. NER и entity-linking (привязка новости к активу)

Это ключевой узел: он отвечает на вопрос «какая новость относится к какому активу».

- **NER** — Natasha (CPU, без torch) извлекает упоминания организаций.
- **Индекс алиасов** строится из БД: тикеры, названия компаний и их синонимы.
- **Матчинг по леммам.** И текст, и алиасы приводятся к начальной форме (лемматизация
  Natasha), затем ищется совпадение **фразы лемм** с границами слова. Это:
  - ловит склонённые формы («Сбербанка», «МосБирже», «Газпрому» → начальная форма);
  - убирает ложные подстроки («газ» внутри другого слова уже не матчится, т.к. сравнение
    идёт по токенам-леммам, а не по подстроке).
  Без морфологии матчер деградирует к простому токенному совпадению.
- **Релевантность связи** (`relevance` ∈ [0, 1]) — уверенность привязки: точный тикер =
  1.0, многословный алиас надёжнее однословного, подтверждение NER-упоминанием повышает
  оценку. Идёт в `ArticleEntity.relevance` и далее в значимость и силу влияния события.

### 1.4. Эмбеддинги (семантический слой)

`intfloat/multilingual-e5-large` (1024-dim, ONNX через fastembed, CPU-friendly). Векторы
хранятся в Postgres через **pgvector** (IVFFlat-индекс, косинусная близость) и дают
семантический поиск похожих новостей для RAG.

## 2. Значимость новости

Модуль `nlp/significance.py`. Единая мера важности ∈ [0, 1] — фундамент для фильтрации,
хранения и алертов.

**Формула** (чистая функция `significance_score`, веса настраиваются `GEO_SIG_W_*`):

```
significance = w_type · type_weight(event_type)      # вес категории события (0.3..1.0)
             + w_sent · |sentiment_score|             # сила тональности
             + w_link · min(1, Σ relevance)           # качество связей с активами
```
По умолчанию `w_type=0.5, w_sent=0.3, w_link=0.2` (сумма = 1 → результат в [0, 1]).
Вес типа: санкции 1.0, отчётность 0.85, M&A 0.8, дивиденды/геополитика 0.7, регуляторика/
макро 0.6, прочее 0.3.

**Опциональный override** — дообученный классификатор значимости
(`GEO_SIGNIFICANCE_ADAPTER_PATH`, бакеты low/medium/high → представительное значение).
Если модель недоступна — используется формула.

Значимость используется в трёх местах:
- **Фильтр на инжесте**: новость типа `other`, без связанных активов и со значимостью
  ниже `GEO_MIN_SIGNIFICANCE` не сохраняется (отсев шума).
- **TTL-хранение** (`storage/retention.py`): срок хранения растёт со значимостью линейно,
  `ttl = round(min_days + (max_days − min_days) · significance)` (по умолчанию 7…365
  дней). Старые малозначимые новости удаляются `geo prune`.
- **Gate алертов**: алерты учитывают только новости со `significance ≥
  GEO_ALERT_MIN_SIGNIFICANCE`.

## 3. Рыночная аналитика

### 3.1. Технические индикаторы

Модуль `analytics/indicators.py` (чистый Python, ряд закрытий старое → новое):
- **SMA** 20/50/200, **EMA**;
- **RSI(14)** по методу Уайлдера;
- **доходности** за 1н/1м/3м (5/21/63 бара);
- **годовая волатильность** = стандартное отклонение дневных доходностей × √252, в %;
- **52-недельный диапазон** (max/min последних 252 баров);
- **тренд** (up/down/flat) — эвристика по расположению цены относительно SMA50/SMA200.

### 3.2. Корреляции с факторами

Модуль `analytics/correlations.py`. Считается **корреляция Пирсона дневных доходностей**
актива с факторами; ряды выравниваются по общим датам, минимум 20 совпадающих точек:
- курс **USD/RUB** (валютный фактор);
- **Brent**, **золото**, **серебро** (сырьевые факторы);
- средняя доходность **пиров по сектору** (отраслевой фактор).

Пирсон реализован на чистом Python. Это даёт честную, проверяемую оценку того, чем
реально движется актив, без «чёрного ящика».

### 3.3. Бэктест сигналов

Модуль `analytics/backtest.py`. Назначение — проверить, давал ли сигнал преимущество на
истории.

**Модель исполнения:** long/flat, без плеча и комиссий (грубая, но честная оценка
направления сигнала). **Без заглядывания в будущее**: позиция на баре t определяется
сигналом, известным на t−1 (`held = [0] + signals[:-1]`).

**Сигналы** (чистые генераторы, возвращают 0/1 на каждый бар):
- `sma_cross` — лонг, пока быстрая SMA выше медленной;
- `momentum` — лонг, если цена выше, чем N баров назад;
- `rsi` — возврат к среднему: вход при перепроданности (<30), выход при >55;
- `sentiment` — новостной: лонг, если суммарный сентимент свежих (за ~5 дней) связанных
  с активом новостей положителен.

**Метрики** (`BacktestResult`): итоговая доходность, доходность buy&hold (бенчмарк),
**CAGR**, **коэффициент Шарпа** (безрисковая ставка 0, годовое масштабирование √252),
**максимальная просадка**, **hit-rate** (доля прибыльных сделок), exposure (доля времени
в рынке), число сделок и кривая капитала (equity curve) для графика.

### 3.4. Свечи и таймфреймы

Данные — дневные OHLC-свечи с MOEX (`prices.ohlc_series`). Для разных горизонтов:
- **зум** по дневным данным (1М/3М/6М/1Г/всё);
- **агрегация** дневных свечей в недельные/месячные на лету (`analytics/resample.py`,
  чистая функция: open — первой свечи, close — последней, high/low — экстремумы периода).
Дашборд рисует линию или японские свечи серверным inline-SVG (`api/charts.py`), без
JS-библиотек.

## 4. Граф факторов и события

### 4.1. Факторы актива

`context/graph.py` — что влияет на актив: сектор и пиры (через компанию-эмитента),
экспертная карта макро-драйверов по секторам (`_SECTOR_MACRO`), явные связи из таблицы
`relations`.

### 4.2. События и оценка влияния

`context/events.py`. Из значимой новости (содержательный тип + связанные активы)
формируется **Event** (одно событие на статью, идемпотентно), а на каждый связанный актив
— **EventImpact**: направление и сила влияния.

**Оценка влияния** (чистая `assess_impact`):
```
magnitude = type_weight(event_type) × max(|sentiment_score|, 0.2) × relevance   # ∈ [0,1]
direction = positive / negative / neutral   # по тональности
```
Это прозрачное правило (тот же вес типа события, что и у значимости), которое позже можно
заменить обученной моделью, сохранив интерфейс.

## 5. Контекст актива и RAG-синтез

`context/asset_context.py` + `query/asset_report.py`. Отчёт по активу — это **RAG**
(retrieval-augmented generation):

1. **Retrieval** — собирается структурированный контекст: индикаторы, макро-снимок,
   факторы, корреляции, топ-события влияния и связанные новости.
2. **Generation** — связный нарратив синтезирует локальный LLM (Qwen2.5-7B через Ollama;
   `nlp/llm.py`, провайдер настраивается, есть cloud-фолбэк). Контекст версионируется в
   таблице `asset_context`. Если LLM недоступен — отдаётся структурированный отчёт без
   связного текста (graceful degradation).

Сводка рынка «что по новостям» (`query/news_summary.py`) устроена так же: макро + топ-
движения + распределение тональности и тем + свежие заголовки, плюс опциональная
LLM-сводка.

## 6. Алерты

`alerts/` — слоистая архитектура: **чистые правила** (`rules.py`) отдельно от
DB-движка (`engine.py`) и доставки (`channels.py`).

Три триггера (все — только по значимым новостям/данным):
- **движение цены** — |Δ close день-к-дню| ≥ порога (≥ 2× порога → critical);
- **всплеск негатива** — по рынку/активу: число негативных новостей ≥ min И доля ≥ ratio;
- **новое значимое событие** — санкции/геополитика → critical.

Каждый алерт несёт детерминированный `dedup_key` (движения/негатив — с дневным bucket,
события — навсегда), поэтому повторный прогон не шлёт дубль (идемпотентность через
уникальный индекс). Доставка: консоль/лог всегда + Telegram (нескольким получателям, с
паузой против троттлинга).

**Управление (UX-слой):** `alerts/manage.py` — подтверждение (`acknowledge` →
`alerts.acknowledged_at`) и подавление (`mute`/`unmute`/`list_mutes` поверх таблицы
`alert_mutes`). Правило подавления адресует тикер, тип алерта или пару `тикер:тип`,
с опциональным сроком (`until`, NULL = бессрочно). Чистая функция `engine._is_muted`
сопоставляет алерт с активными правилами; в `evaluate_and_dispatch` между записью и
доставкой замьюченные **сохраняются** (история не теряется), но не рассылаются
(`channels=["muted"]`). Веб-UI — `/ui/alerts` (лента с фильтрами + ack/mute/unmute),
выборка через `query/alerts_feed.recent_alerts` (фильтры severity/type/ticker/unacked).

## 6a. Текстовый интерфейс к аналитике (RAG-оркестратор)

`query/ask.py` — свободный вопрос → ответ поверх УЖЕ готовых query-функций (ничего не
пересчитывает). Поток: **интент** (LLM по строгому промпту → JSON `{intent, ticker,
hours, strategy, query}`, чистый `_parse_intent`) → **диспатч** в существующее
(`build_report`/`build_snapshot`/`backtest_asset`/`recent_events`/`recent_alerts` или
`semantic.search_news`) → **нарратив** (короткий заземлённый ответ LLM по собранным
фактам + цитаты-ссылки). Интенты: `asset|market|backtest|events|alerts|news`.

`query/semantic.py` — pgvector-поиск (`Embedding.vector.cosine_distance`, IVFFlat-индекс),
**первый потребитель** эмбеддингов из `embeddings` (заполняются при инжесте в M1). Запрос
эмбеддится тем же путём, что и статьи (`embed_one`, без e5-префиксов → пространства
консистентны).

**Graceful-фолбэк** (как везде): LLM недоступен/интент невнятный → чистая эвристика
`_fallback_intent` (тикер по `EntityIndex`, ключевые слова), нарратив → шаблон из фактов.
UI — ask-бокс на дашборде (`/ui/partials/ask`, HTMX + GET-фолбэк без JS).

## 7. Дообучение под домен

`nlp/dataset.py`, `scripts/build_dataset.py`, `scripts/llm_label.py`, `scripts/train_lora.py`.
Только маленькие encoder-модели дообучаются под домен; большой LLM не обучается. Дообучение —
на машине пользователя (extras `[train]`: peft/transformers/datasets/accelerate), на CPU.

### 7.1. Дистилляция LLM-учителя → полный fine-tune (используется для сентимента)

Рабочий рецепт для сентимента. Решает проблему качества «в лоб»:

1. **Золотая разметка учителем** (`scripts/llm_label.py`): локальный Qwen2.5 (тот же, что
   и для синтеза) размечает новости из БД на тональность с точки зрения инвестора. Скрипт
   идемпотентен и резюмируем (пишет JSONL инкрементально, при перезапуске пропускает
   размеченное) — CPU-генерация долгая. Получается «золото» с реалистичным распределением
   классов (живой positive), а не вырожденное «серебро».
2. **Полный fine-tune** (`scripts/train_lora.py --full-finetune`): дообучается **весь**
   небольшой энкодер (`rubert-tiny2`, 29М — минуты на CPU), модель сохраняется целиком
   (config + веса + `labels.json`), а не как адаптер.
3. **Подключение**: `GEO_SENTIMENT_ADAPTER_PATH` → каталог полной модели; `nlp/sentiment.py`
   распознаёт полную модель (нет `adapter_config.json`) и грузит напрямую.
4. **Переразметка истории**: новая модель применяется к новым новостям на инжесте, но
   исторические статьи хранят метки старой модели. `geo rescore` (`processing.rescore_existing`)
   приводит их в соответствие — пересчитывает выбранные модельные поля (`--what sentiment,
   events,significance`) по уже сохранённым тексту и связям, синхронизирует денормализованную
   тональность в `article_entities` и пересчитывает производную значимость. Идемпотентно,
   батчами, с `--dry-run` (показывает сдвиг распределения, ничего не записывая). Это общая
   точка входа для любой будущей смены модели, а не разовый скрипт.

**Почему не LoRA для сентимента.** LoRA поверх *замороженной* базы для дистилляции **не
работает**: признаки базовой `blanchefort` заточены под её собственный (общеязыковой)
сентимент и расходятся с разметкой учителя. Линейный пробинг по замороженным `[CLS]` и LoRA
любой ёмкости упираются в качество уровня случайного угадывания, а выход коллапсирует в
константу. Перекрыть это можно только адаптацией самого энкодера — то есть полным fine-tune.

### 7.2. LoRA + weak supervision (исходный механизм M4)

Облегчённый путь, оставлен для классификаторов, где базовых/правиловых меток достаточно:

- **Weak supervision**: текущие (правиловые/базовые) предсказания конвейера используются
  как «серебряные» метки. **LoRA** меняет доли процента весов (дёшево на CPU).
- На практике для событий/значимости weak-supervision LoRA-адаптеры пока **слабее правил**,
  поэтому в проде они не подключены (тип события — на регулярных правилах, значимость — на
  формуле). Кандидат на ту же дистилляцию учителем, что и сентимент.

Во всех случаях `GEO_*_ADAPTER_PATH` опциональны: без них — graceful-фолбэк на
правила/формулу/базовую модель. Общий загрузчик LoRA-адаптеров — `nlp/_seqcls.py`.

## 8. Допущения и ограничения (важно для интерпретации)

- **Дневные данные.** Внутридневных свечей нет — все индикаторы/бэктест на дневных
  закрытиях; недельный/месячный таймфрейм получается агрегацией.
- **Бэктест идеализирован**: long/flat, без комиссий, проскальзывания и плеча — это оценка
  *направления* сигнала, а не торговая стратегия «как есть».
- **Корреляция ≠ причинность.** Корреляции Пирсона показывают совместное движение, а не
  механизм влияния.
- **Серебряные метки шумны.** Значимость/влияние событий — прозрачные эвристики; их
  качество ограничено качеством сентимента и линковки (поэтому и заложено дообучение).
- **CPU-ограничения.** LLM-синтез медленный и опциональный; основной поток аналитики
  работает и без него.
- **Источники могут троттлиться/отваливаться** — конвейер устойчив (ретраи,
  идемпотентность по уникальным ключам, обработка по одному документу в SAVEPOINT).

## Реестр аналитических метрик (что считаем / что осознанно пропускаем)

**Во времени (персистится):** новостные исходы (`news_outcomes`: ret/abn 1/3/5д), индекс
настроения (`market_sentiment`: mean/EWMA/breadth/dispersion по рынку/сектору/активу — B1),
снимки портфеля (`portfolio_snapshots`), прогоны оценки (`eval_runs`).

**Считается на лету (не персистится):** теханализ (RSI/MACD/Bollinger/SMA/EMA/ATR/Stoch/OBV),
волатильность (EWMA RiskMetrics), режимы рынка (HMM), факторная атрибуция (OLS, betas/R²),
корреляции, новостное давление, граф-импакты соседей, риск-метрики портфеля (VaR/Sharpe/
вклад в риск по Эйлеру), бэктест-метрики (Sharpe/Sortino/Calmar/PF).

**Где метрики ВЛИЯЮТ на ответ:** теханализ (RSI/MACD/Bollinger/тренд) + тональный моментум/
ширина/дивергенция (B1) → грунт LLM-нарратива (`grounding.py`, асет-карточка и `ask`);
консенсус настроения (стойка/перекос секторов/дивергенции) → сводка рынка (`news_summary`).

**Осознанно НЕ считаем (нет данных/вне скоупа):** ликвидность/стаканы (нет L2),
опционные поверхности (нет опционных данных), факторы Фамы-Френча (нет факторных портфелей),
интрадей-микроструктура. Эти пробелы — не баг, а отсутствие источника.
