# geoanalytics

Инструмент для аналитики мировой геополитики и экономики **с упором на рынок РФ**.
Парсит свежие новости, отслеживает котировки и макропоказатели, хранит контекст по
каждому активу и учитывает внешние факторы (макро, геополитика, глобальные тренды).

Запросы пользователя:
- **что по новостям** → сводка текущей ситуации (индексы, курс, ключевая ставка, топ-движения, настроение, главные заголовки);
- **анализ актива X** → показатели актива + связанная мировая обстановка + перспективы.

Источники новостей: **Интерфакс**, **РБК**, **Ведомости**, **Коммерсантъ** (RSS);
рынок — **МосБиржа (ISS)**; макро — **ЦБ РФ**, **ФРС (FRED)**, **ЕЦБ (ECB)**, нефть Brent.
Список расширяемый.

## Архитектура (кратко)

Слои: `connectors` (парсинг) → `storage` (Postgres/Timescale/pgvector) →
`nlp` (NER, сентимент, эмбеддинги, LLM) → `analytics` (индикаторы) →
`context` (контекст актива + граф связей) → `query` (роутер интента, RAG) с
оркестрацией через `orchestration`.

**Модели — локальный каскад** маленьких специализированных моделей (эмбеддинги,
NER, сентимент, классификация), тяжёлый синтез — один 7–8B LLM через Ollama.

Полный план: `/home/ijstt/.claude/plans/fizzy-seeking-mountain.md`.

## Быстрый старт (M0/M1)

```bash
# 1. Зависимости (рекомендуется uv)
uv venv && source .venv/bin/activate
uv pip install -e ".[nlp,dev]"

# 2. Поднять БД и Ollama
cp .env.example .env
docker compose up -d

# 3. Применить миграции
geo db upgrade

# 4. Справочник эмитентов и история котировок
geo db seed
geo backfill            # история с MOEX для индикаторов

# 5. Собрать + обработать данные и посмотреть сводку
geo pipeline
geo news
geo asset SBER
```

## Запуск / остановка одной командой

После первичной установки всей системой управляет `geo-ctl.sh` (контейнеры БД+Ollama и
службы алертов+дашборда вместе):

```bash
./geo-ctl.sh up        # поднять всё
./geo-ctl.sh down      # остановить всё (данные в volume сохраняются)
./geo-ctl.sh restart   # перезапустить
./geo-ctl.sh status    # что работает (контейнеры, службы, /health)
./geo-ctl.sh logs      # живой хвост логов планировщика
```

Контейнеры помечены `restart: unless-stopped`, а docker daemon включён в автозапуск —
**после перезагрузки машины стек поднимается сам**, ничего вручную делать не нужно.
Первичная установка (миграции/сид/история) — отдельный «тяжёлый» bootstrap `./up.sh`.

Службы главной машины под `geo-ctl.sh`: `geo-alerts` (scheduler), `geo-dashboard`, `geo-bot`,
`geo-depth`. Набор переопределяется через `GEO_SERVICES`.

### Торговый процесс `geo-futrader` (Трек 2) — на отдельном хосте (Raspberry Pi)

Дневная futrader-петля (accumulate → train → eval → paper, + еженедельные PBO/drift) и
интрадей-цикл бумажной торговли вынесены из scheduler в **отдельный процесс `geo run-futrader`**.
Это снимает с scheduler бывший ~8G-пик памяти. Трейдер — чисто numeric (sklearn/индикаторы), без
GPU/LLM/Ollama; он только читает `market_regimes`/`market_sentiment` из Postgres главной машины
(их пишет scheduler) и пишет в `futures_*` там же.

Поэтому он живёт на **Raspberry Pi 4 (8GB) в локальной сети** как один писатель `futures_*` —
переносится без правок кода (указанием `GEO_DB_HOST` на главную машину). На главной машине
трейдер не запускается. **Инвест-каскад (3 трансформера + e5-эмбеддер + Ollama) остаётся на
главной машине с GPU.** Полный runbook переноса — `deploy/pi/README.md`.

## Команды CLI

| Команда | Назначение |
|---|---|
| `geo sources` | Список доступных источников |
| `geo ingest [--source ...]` | Парсинг источников в raw-слой |
| `geo process` | Обработка raw → новости/котировки/макро (NER, сентимент, эмбеддинги) |
| `geo pipeline [--source]` | Полный цикл: ingest + process |
| `geo news [--hours N] [--no-llm]` | Сводка «что по новостям» |
| `geo asset TICKER [--no-llm]` | Аналитика актива: индикаторы, макро, факторы, корреляции, события, новости, контекст |
| `geo events [--hours N]` | Значимые события и их влияние на активы |
| `geo backtest TICKER [-S strategy]` | Бэктест сигнала (sma_cross / momentum / rsi / sentiment) |
| `geo backfill [-t TICKER] [-d DAYS] [--fx]` | Загрузка истории котировок с MOEX (`--fx` — курсы ЦБ) |
| `geo calendar [--sync] [-d DAYS]` | Календарь событий: заседания ЦБ, дивидендные отсечки (вкл. будущие) |
| `geo attribution TICKER [-d DAY]` | Факторное разложение дня: рынок/сектор/FX → идиосинкразия |
| `geo regime [-s 2\|3]` | Режим рынка (HMM по волатильности IMOEX и USD/RUB) |
| `geo retemporal [-n N]` | Переразметка статей temporal-моделью (статус + дата события) |
| `geo renumeric [-n N]` | Извлечение чисел из старых статей: дивиденд/ставка/сумма сделки (F5) |
| `geo context TICKER` | Пересборка контекста актива |
| `geo relink` | Перелинковать сущности по уже сохранённым новостям (после установки NLP-моделей) |
| `geo db upgrade` / `geo db seed` | Миграции / справочник эмитентов |
| `geo serve [--port N] [--reload]` | Запустить REST API (FastAPI), документация на `/docs` |
| `geo alerts [--no-dispatch]` | Проверить триггеры алертов, разослать новые, показать ленту |
| `geo run-scheduler` | Периодический сбор данных (включает проверку алертов) |
| `geo run-futrader` | Трек 2: автономный торговый цикл (отдельная служба, чисто numeric, Pi-ready) |

## Дообучение моделей (опционально)

Маленькие encoder-модели можно дообучить под домен. Подробно о подходах — в
`ANALYTICS.md` §7. Рабочий рецепт для сентимента — **дистилляция LLM-учителя** (Qwen2.5
размечает «золото», затем полный fine-tune небольшого энкодера; LoRA-поверх-frozen для
сентимента не работает):

```bash
pip install -e ".[train]"                              # peft, datasets, accelerate
python scripts/llm_label.py --task sentiment \
    --out data/sentiment_gold.jsonl                    # «золотая» разметка локальным Qwen2.5
python scripts/train_lora.py --task sentiment --full-finetune \
    --base cointegrated/rubert-tiny2 \
    --dataset data/sentiment_gold.jsonl \
    --output data/adapters/sentiment-ft-tiny2          # полный fine-tune (сохраняет модель целиком)
export GEO_SENTIMENT_ADAPTER_PATH=data/adapters/sentiment-ft-tiny2  # включить в проде
```

LoRA по «серебряным» меткам (исходный путь M4, `scripts/build_dataset.py` →
`scripts/train_lora.py` без `--full-finetune`) остаётся для классификаторов событий/
значимости. `GEO_*_ADAPTER_PATH` понимает и LoRA-адаптер, и полную модель.

## Веб-интерфейс и API (M5)

```bash
pip install -e ".[api]"
geo serve                       # http://127.0.0.1:8800  (дашборд: /, Swagger: /docs)
```

**Дашборд (HTMX + Jinja, server-rendered, без сборки):**

| Страница | Назначение |
|---|---|
| `/` | Сводка рынка + **ask-бокс** «спросить аналитику» (макро, топ-движения, настроение, заголовки; тикеры кликабельны) |
| `/ui/asset?ticker=SBER` | Отчёт по активу + SVG-график цены (пиры/новости кликабельны) |
| `/ui/backtest?ticker=SBER&strategy=rsi` | Бэктест + кривая капитала |
| `/ui/alerts` | Лента алертов с фильтрами + подтверждение (ack) и подавление (mute) |

**REST API** (та же логика, тонкая обёртка над слоем `query`):

| Метод | Назначение |
|---|---|
| `GET /health` | Живость + число источников |
| `GET /sources` | Список источников |
| `GET /news?hours=24` | Сводка «что по новостям» |
| `GET /asset/{ticker}` | Аналитический отчёт по активу (404, если нет) |
| `GET /backtest/{ticker}?strategy=rsi` | Бэктест сигнала |
| `GET /events?hours=168` | События и их влияние на активы |
| `GET /alerts?hours=168` | Лента сработавших алертов |

## Алерты (M5.3)

Триггеры поверх собранных данных, с дедупликацией (одно уведомление на срабатывание):

- **движение цены** — актив изменился на ≥ `GEO_ALERT_PRICE_PCT`% за день;
- **всплеск негатива** — негативных новостей по активу/рынку ≥ порога и доля ≥ `GEO_ALERT_NEG_RATIO`;
- **новое событие** — появилось значимое событие (санкции, отчётность, дивиденды…).

```bash
geo alerts                      # проверить триггеры, разослать новые, показать ленту
geo alerts --no-dispatch        # только зафиксировать, без уведомлений
```

Доставка — всегда в лог, плюс **Telegram**, если заданы `GEO_TELEGRAM_BOT_TOKEN` и
`GEO_TELEGRAM_CHAT_ID`. `geo run-scheduler` проверяет алерты в каждом цикле.

**Управление в вебе (`/ui/alerts`):** лента фильтруется по уровню/типу/тикеру и
«только непросмотренные»; кнопка **✓** подтверждает алерт (ack); **подавление (mute)**
создаёт правило по тикеру, типу или паре `тикер:тип` (на N дней или бессрочно).
Замьюченные алерты по-прежнему **фиксируются** в БД (история не теряется), но не
рассылаются по каналам (`channels=["muted"]`). Фильтр подавления применяется в
`evaluate_and_dispatch` между записью и доставкой.
